# Golden Dataset Collection Guide — Sprint 001

> *Первый датасет должен учить систему ошибаться правильно.*

---

## 1. Dataset Purpose

### 1.1 Для чего этот датасет

Этот датасет **не предназначен** для обучения поиска, ранжирования или категоризации.

Он предназначен для одного вопроса:

> **Может ли система восстановить реальных продавцов из разрозненных объявлений?**

Entity Resolution — это проверка «кто есть кто» на рынке. Без ответа на этот вопрос любой собранный массив данных — просто шум.

### 1.2 Что мы узнаем из датасета

| Вопрос | Проверяется группой |
|--------|-------------------|
| Работает ли телефон как primary key рынка | A, B, C |
| Можно ли доверять адресу | B, C |
| Создают ли шаблонные тексты ложные совпадения | C, D |
| Как часто фото копируются между продавцами | B, D |
| Какой процент объявлений не содержит контактов | D |

## 2. Collection Strategy

### 2.1 Структура выборки

50 объявлений делятся на четыре группы.

```
Структура датасета:

Группа A: Явные совпадения        14 объявлений  (28%)
Группа B: Сложные совпадения      14 объявлений  (28%)
Группа C: Похожие конкуренты      14 объявлений  (28%)
Группа D: Неопределённые случаи    8 объявлений  (16%)
```

### 2.2 Группа A: Явные совпадения (14 шт.)

Один продавец → несколько объявлений. Система **должна** их объединить.

```yaml
group_a:
  purpose: "Проверить базовую способность системы объединять очевидные случаи"
  
  requirements:
    - один продавец, 2–4 объявления
    - один и тот же телефон
    - один и тот же адрес (или очевидная связь)
    - минимум 4 разных продавца
    - хотя бы один с 2+ площадками
  
  expected_confidence: "≥ 0.85"
  expected_difficulty: "low"
  
  examples:
    - "ИП Иванов" на Avito и "ИП Иванов" на Юле — один телефон
    - Частное лицо продаёт ноутбук + телефон + планшет — один номер
    - Компания с 5 разными товарами — один адрес, один сайт
```

### 2.3 Группа B: Сложные совпадения (14 шт.)

Один продавец → разные названия / разный текст. Система должна выявить скрытую связь.

```yaml
group_b:
  purpose: "Проверить способность видеть продавца под разными масками"
  
  requirements:
    - один продавец, 2–4 объявления
    - разные названия в каждом объявлении
    - разный стиль описания (но фактические признаки совпадают)
    - минимум 3 разных продавца
    - хотя бы один случай с копированными фото
  
  expected_confidence: "≥ 0.65"
  expected_difficulty: "medium"
  
  examples:
    - "РемСервис" на Avito, "ИП Иванов" на Юле, "Холод-Мастер" на VK — один телефон
    - Частное лицо с одним телефоном, разными именами
    - Перекупаем: один телефон, 10 товаров, разные имена (Иван, Сергей, Дмитрий)
```

### 2.4 Группа C: Похожие конкуренты (14 шт.)

Разные продавцы → похожий текст / одинаковые категории. Система **НЕ должна** их объединять.

```yaml
group_c:
  purpose: "Проверить устойчивость к ложным срабатываниям"
  
  requirements:
    - 2+ продавца в одной категории
    - похожий или шаблонный текст у всех
    - одинаковые бренды в ассортименте
    - разные телефоны
    - минимум 3 группы конкурентов
    - хотя бы один случай с общим адресом (ТЦ)
  
  expected_confidence: "≤ 0.40"  # система не должна быть уверена
  expected_difficulty: "medium"
  
  examples:
    - 5 мастеров по ремонту холодильников — все пишут «Срочно! Дорого!» 
    - 3 продавца iPhone — одинаковый шаблон «не бит, не крашен»
    - Магазины в одном ТЦ — разные названия, один адрес, разные телефоны
```

### 2.5 Группа D: Неопределённые случаи (8 шт.)

Недостаточно данных для уверенного решения.

```yaml
group_d:
  purpose: "Проверить, умеет ли система говорить «я не знаю»"
  
  requirements:
    - объявления без телефона
    - объявления без адреса
    - только имя и город
    - минимум 3 таких случая
  
  expected_confidence: "≤ 0.30"
  expected_difficulty: "high"
  
  examples:
    - "Продам iPhone" — только имя, нет контактов
    - "Услуги сантехника" — только город
    - "Срочно! Недорого!" — нет никаких данных
```

## 3. Source Diversity

### 3.1 Распределение по источникам

```yaml
sources:
  avito:
    count: 20
    purpose: "Основной источник классифайдов"
  
  youla:
    count: 15
    purpose: "Вторичный рынок, перекрёстные совпадения"
  
  vk_market:
    count: 10
    purpose: "Социальная сеть как площадка"
  
  telegram:
    count: 5
    purpose: "Каналы и чаты объявлений"
```

### 3.2 Почему именно такое распределение

- Avito — крупнейший источник, основа выборки
- Юла — для кросс-площадочных совпадений (один продавец на двух площадках)
- VK — проверка коннектора к социальным сетям
- Telegram — самый «грязный» источник, минимальный объём

### 3.3 Требование к разнообразию

```yaml
diversity_rules:
  min_sources: 3                    # минимум 3 разных источника
  max_from_one_source: 20           # не более 20 из одного
  cross_platform_pairs: 5           # минимум 5 пар с разных площадок
  single_platform_only: true        # разрешены объявления только с одной площадки
```

## 4. Privacy Rules

### 4.1 Что можно хранить

```yaml
store_allowed:
  - название объявления
  - цена
  - описание (обезличенное — без номеров заказов)
  - категория
  - город / регион
  - адрес (без квартиры)
  - фотографии (без лиц, без номеров на фото)
  - ссылка на объявление
  - дата публикации
```

### 4.2 Что маскировать

```yaml
mask_required:
  phone:
    rule: "Последние 4 цифры открыты, остальное маскируется"
    example: "+7 (***) ***-45-67"
  
  email:
    rule: "Домен открыт, локальная часть маскируется"
    example: "i*******@gmail.com"
  
  seller_name:
    rule: "Имя и фамилия — первые буквы"
    example: "Иванов С. А." → "Иванов С. А." (можно оставить, если это публичные данные)
```

### 4.3 Для тестового датасета

Внутренний датасет для entity resolution использует полные данные (телефоны, адреса). Датасет **не публикуется** в открытом доступе. Хранится в защищённой директории проекта.

Если в будущем датасет будет опубликован — применяются правила маскировки из п. 4.2.

## 5. Quality Check

### 5.1 Обязательные поля

Каждое объявление проверяется по чек-листу перед добавлением в датасет:

```yaml

  - field: source
    required: true
    format: "avito | youla | vk | telegram"
  
  - field: url
    required: true
    format: "полная ссылка"
  
  - field: title
    required: true
    format: "как в объявлении"
  
  - field: category
    required: true
    format: "двухуровневая"
  
  - field: seller_name
    required: true
```

## 6. Human Annotation Rules

### 6.1 Основные правила

```yaml
annotation_rules:
  rule_1:
    name: "Не угадывать"
    description: "Если нет доказательств — ставь Unknown"
    example: "Только имя и город → Unknown (а не 'возможно, тот же')"
  
  rule_2:
    name: "Доказательства выше интуиции"
    description: "Даже если кажется, что 'похожи' — без фактов не объединять"
    example: "Два объявления о ремонте холодильников, разные телефоны, разные адреса → Different"
  
  rule_3:
    name: "Каждая связь — с объяснением"
    description: "Почему два объявления — один продавец?"
    required: ["reason", "evidence", "confidence"]
  
  rule_4:
    name: "Сложные случаи — на обсуждение"
    description: "Неуверен? Отмечай и обсуждай со вторым разметчиком"
    action: "flag_for_review: true"
```

### 6.2 Формат аннотации

```yaml
annotation_entry:
  listing_id: "avito_001"
  
  human_judgment:
    belongs_to_seller: "SELLER_003"   # или null
    confidence: 0.95                    # 0.0–1.0
    reason: "Телефон + адрес совпадают с youla_007"
    evidence:
      - type: phone
        value: "+7 (***) ***-45-67"
        source_listing: "youla_007"
      - type: address
        value: "ул. Ленина, 10"  
        source_listing: "youla_007"
    flagged: false
```

### 6.3 Уровни уверенности разметчика

```yaml
annotator_confidence:
  0.90–1.00: "Абсолютно уверен — несколько независимых доказательств"
  0.70–0.90: "Уверен — есть доказательства, но не все проверены"
  0.50–0.70: "Предположение — есть признаки, но недостаточно"
  0.00–0.50: "Не знаю — отметь для обсуждения"
```

## 7. Final Acceptance Criteria

### 7.1 Критерии готовности датасета

```yaml
acceptance:
  
  # Количество
  total_listings: 50
  min_seller_groups: 10
  
  # Структура
  group_a_count: { min: 10, max: 16 }
  group_b_count: { min: 10, max: 16 }
  group_c_count: { min: 10, max: 16 }
  group_d_count: { min: 5, max: 10 }
  
  # Качество
  fields_complete: 100%           # все обязательные поля заполнены
  urls_working: > 90%             # ссылки активны на момент сбора
  no_duplicates: true             # нет повторяющихся объявлений
  
  # Аннотация
  seller_groups_identified: true  # ground truth определён
  cross_annotator_agreement: > 0.80  # два разметчика согласны
  evidence_documented: true       # каждая связь имеет evidence
  
  # Сложные случаи
  has_cross_platform: true        # есть объявления с разных площадок
  has_ambiguous: true             # есть неопределённые случаи
  has_conflicts: true             # есть конкуренты в одной категории
```

### 7.2 Процесс утверждения

```yaml
approval:
  step_1: "Сборщик проверяет чек-лист"
  step_2: "Второй разметчик проверяет спорные случаи"
  step_3: "Оба согласны — датасет утверждается"
  step_4: "Датасет сохраняется в golden_dataset_v1/"
  step_5: "Начинается Entity Resolution Test #001"
```

## 9. Final Principle

> **Первый датасет должен учить систему ошибаться правильно.**

Ошибки неизбежны. Но если система ошибается предсказуемо — мы знаем, что чинить. Если ошибки хаотичны — у нас нет датасета, у нас есть шум.

50 качественных размеченных объявлений ценнее 50 000 сырых записей, потому что они дают первый ориентир — **что такое «правильно» для этой системы**.
