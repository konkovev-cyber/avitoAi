# Market Intelligence OS — Constitution v1.0

> *Мы не собираем объявления. Мы строим доказательную карту реального рынка.*

---

## 1. Purpose

### 1.1 Почему система существует

Система создана для одной цели: **выявления реальных экономических сущностей** — продавцов, компаний, связей, возможностей — на основе разрозненных публичных данных.

### 1.2 Чем система НЕ является

- ❌ Парсером объявлений
- ❌ Агрегатором ссылок
- ❌ Поисковой строкой
- ❌ Монитором конкретной площадки

### 1.3 Чем система ЯВЛЯЕТСЯ

- ✅ **Market Intelligence Platform** — платформа разведки рынка
- ✅ **Evidence Engine** — система, где каждый вывод подтверждён доказательствами
- ✅ **Entity Graph** — граф экономических сущностей, а не список ссылок

---

## 2. Core Model

### 2.1 Основной поток данных

```
Source                          — источник данных (Avito, Юла, сайт, Telegram)
  ↓
Listing                         — сырое объявление (временное свидетельство)
  ↓
Evidence                        — извлечённые признаки (телефон, адрес, фото...)
  ↓
Entity Resolution               — объединение по совокупности доказательств
  ↓
Seller / Company                — реальная экономическая сущность
  ↓
Market Graph                    — граф связей между сущностями
  ↓
Opportunity Engine              — выявление рыночных возможностей
```

### 2.2 Роль каждой сущности

| Сущность | Роль | Статус |
|----------|------|--------|
| **Source** | Определяет происхождение данных и уровень доверия к ним | Всегда сохраняется |
| **Listing** | Сырое свидетельство — доказывает, что предложение существовало | Никогда не удаляется |
| **Evidence** | Извлечённый признак (телефон, email, адрес, хеш фото) | Основа всех решений |
| **Seller** | Предполагаемый продавец — результат объединения evidence | Вероятностный |
| **Company** | Юридическое лицо — подтверждённая организация | Требует верификации |
| **Market Graph** | Карта связей: кто, где, как, с кем связан | Постоянно уточняется |
| **Opportunity** | Вывод системы о незакрытой потребности рынка | Итоговый продукт |

## 3. Evidence First Principle

### 3.1 Главный закон

> **Ни один вывод без доказательства.**

Каждое утверждение системы должно содержать:

```yaml
statement: "Listing A и Listing B принадлежат одному продавцу"
evidence:
  - type: phone_match
    value: "+7 (999) 123-45-67"
    weight: 0.35
    source: avito_listing_123, avito_listing_456
    timestamp: 2026-07-16T14:00:00Z
  - type: photo_hash_match
    value: "sha256:a1b2c3..."
    weight: 0.20
    source: youla_listing_789
confidence: 0.55
decision: POSSIBLE_SAME_SELLER
requires_review: true
```

### 3.2 Evidence Chain

Каждое звено цепочки должно быть прослеживаемо:

```
Вывод
  ↑
  └─ Evidence #1: совпадение телефона (вес 0.35, источник Avito)
  └─ Evidence #2: совпадение логотипа (вес 0.20, источник Юла)
  └─ Evidence #3: совпадение адреса (вес 0.15, источник сайт компании)
  └─ Negative Evidence: разные города (вес -0.10)
```

### 3.3 Запрещено

- ❌ Делать вывод без указания источника
- ❌ Использовать признаки без весов
- ❌ Скрывать противоречия

## 4. Entity Resolution Policy

### 4.1 Система НЕ имеет права

| Запрет | Обоснование |
|--------|-------------|
| Объединять сущности по одному признаку | Телефон может быть общим, контактное лицо — разным |
| Считать название компании уникальным идентификатором | ООО «Альфа», ИП Иванов, «Техника+» — может быть одним лицом |
| Удалять сомнительные данные | Сомнительные данные — это evidence, а не шум |
| Автоматически мержить без confidence score | Каждое объединение должно быть объяснимо |

### 4.2 Система должна

- ✅ Сохранять все варианты (не перезаписывать, а добавлять)
- ✅ Хранить вероятность (confidence никогда не равен 1.0 без верификации)
- ✅ Объяснять решение (текстовая цепочка рассуждений)
- ✅ Разрешать конфликты (новые данные могут противоречить старым)

### 4.3 Ключевой принцип

> **False Merge хуже False Split.**

Лучше создать двух продавцов и потом объединить, чем ошибочно объединить конкурентов.

## 5. Confidence Framework

### 5.1 Уровни уверенности

| Диапазон | Уровень | Действие AI | Требуется верификация |
|----------|---------|-------------|----------------------|
| 0–40% | **Unknown** | Сохранить как гипотезу | Всегда |
| 40–70% | **Possible Match** | Предложить как вариант | Да |
| 70–90% | **Strong Candidate** | Использовать с пометкой | Да, выборочно |
| 90–98% | **High Confidence** | Использовать в анализе | Нет, но audit trail обязателен |
| 98%+ | **Verified Entity** | Считать установленным | Только при пересмотре |

### 5.2 Правила принятия решений

```yaml
auto_merge_threshold: 0.90    # выше этого — AI объединяет автоматически
suggest_merge_threshold: 0.70 # выше этого — предлагается человеку
human_review_required: 0.50   # выше этого — отмечается для проверки
no_action_below: 0.40         # ниже этого — только гипотеза
```

### 5.3 Объяснение confidence

Каждый score должен сопровождаться:

- **Разложением весов** — какие признаки дали какой вклад
- **Negative evidence** — какие признаки противоречат
- **Reasoning chain** — текстовое объяснение логики

## 6. Data Governance

### 6.1 Неизменяемость истории

- Listing никогда не удаляется — только помечается как `inactive`
- Evidence никогда не перезаписывается — только добавляется
- Изменение entity создаёт новую версию, не меняя старую

### 6.2 Audit Trail

Каждое действие системы фиксируется:

```yaml
action: entity_merge
timestamp: 2026-07-16T14:00:00Z
actor: ai_entity_resolver_v1
input_entities: [seller_123, seller_456]
output_entity: seller_789
confidence: 0.85
evidence_ids: [ev_abc, ev_def, ev_ghi]
reason: "3 of 5 evidence points match"
```

### 6.3 Версии данных

- Каждая версия модели данных имеет номер
- Миграции обратно совместимы
- Старые данные не переформатируются — только дополняются

### 6.4 Исправление ошибок

- Ошибка не удаляется — создаётся correcting evidence
- Исправление имеет свой confidence и обоснование
- Пользователь может оспорить решение системы

## 7. AI Role

### 7.1 AI может

| Действие | Условие |
|----------|---------|
| Искать связи между объявлениями | Всегда |
| Строить гипотезы о принадлежности | С указанием confidence |
| Расширять поисковые запросы | Автоматически |
| Анализировать рынок | На основе подтверждённых данных |
| Предлагать объединение сущностей | На основе evidence |
| Оценивать качество данных | Автоматически |

### 7.2 AI НЕ может

| Запрет | Причина |
|--------|---------|
| Скрывать неопределённость | Это вводит в заблуждение |
| Создавать факты | AI строит гипотезы, а не факты |
| Удалять доказательства | Evidence — основа системы |
| Выдавать предположение за истину | Требуется confidence score |
| Автоматически объединять без порога | Только при confidence ≥ 0.90 |

## 8. Connector Principle

### 8.1 Единый интерфейс

Любой источник данных — Avito, Юла, VK, Telegram, сайт компании, каталог — преобразуется в единую модель Listing.

### 8.2 Изоляция

- Ядро системы **не знает** источник данных
- Вся бизнес-логика работает с унифицированной моделью
- Connector отвечает только за: `connect()` → `search()` → `normalize()` → `Listing`

### 8.3 Абстракция

```yaml
connector:
  source: avito
  type: marketplace
  trust_level: medium
  rate_limit: 10/minute
  features:
    - search_by_query
    - search_by_category
    - get_details
```

## 9. Market Graph Principle

### 9.1 Главный результат

Система не возвращает список объявлений. Она строит граф:

```
Seller
 ├── Listings (n)
 ├── Contacts (телефоны, email, мессенджеры)
 ├── Companies (юрлица, ИП)
 ├── Categories (товары/услуги)
 ├── Locations (адреса, регионы)
 ├── Relationships (связи с другими продавцами)
 ├── Channels (площадки присутствия)
 └── Evidence Chain (доказательства каждого вывода)
```

### 9.2 Граф — живая структура

- Вершины добавляются, но не удаляются
- Рёбра имеют вес (confidence) и направление
- Граф уточняется с каждым новым источником

## 10. Opportunity Engine

### 10.1 Будущее развитие

После построения карты рынка система может выявлять:

| Возможность | Пример |
|-------------|--------|
| Незакрытые потребности | 70% продавцов без сайта |
| Слабые игроки | Медленный ответ на заявки |
| Рыночные ниши | Никто не предлагает услугу в регионе |
| Потенциальные клиенты | Компании с устаревшим оборудованием |
| Точки автоматизации | Повторяющиеся задачи у продавцов |

### 10.2 Принцип

Opportunity не создаётся AI — AI находит закономерности. Решение о том, является ли закономерность возможностью, принимает человек.

## 11. Failure Modes

### 11.1 Чего система должна избегать

| Режим отказа | Признак | Предотвращение |
|-------------|---------|----------------|
| **Шум вместо информации** | Confidence < 40%, но данные показываются как факт | Фильтр по confidence |
| **Красивые дашборды без качества** | Метрики растут, но бизнес-ценность нулевая | Метрики качества данных |
| **Ложные связи** | Объединение конкурентов в одну сущность | False Merge хуже False Split |
| **Отсутствие объяснений** | AI сказал «это один продавец» без цепочки | Evidence First Principle |
| **Автоматические решения без контроля** | Система сама решила, без human review | Пороги confidence |

### 11.2 Мониторинг качества

```yaml
quality_metrics:
  false_positive_merge_rate: < 1%
  false_negative_merge_rate: < 10%
  evidence_coverage: > 90%
  confidence_calibration: tracked
```
